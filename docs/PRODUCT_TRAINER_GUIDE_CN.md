# WorkBuddy OSS 产品全景讲解与用户培训稿

版本口径：WorkBuddy OSS `v1.1.16`  
适用对象：企业管理者、业务负责人、客服、销售、运营、HR、IT 管理员、实施人员和开源用户

## 一段话介绍

WorkBuddy OSS 是一套面向中国企业团队、支持私有化部署的开源 IM Agent 工作流中台。它不是一个只负责聊天的机器人，也不是另一个通用 AI 应用搭建平台，而是把飞书、企业微信、钉钉、Webhook、CSV 或 JSON 中分散的聊天消息，自动整理成工单、销售线索、跟进任务、候选人、知识条目和运营报告。AI 负责识别意图、判断风险、生成业务对象和回复草稿，人负责审批、修改和执行，系统负责记录全过程。它最终要解决的问题，是让企业里原本散落在聊天窗口中的客户问题、销售机会、社群风险、招聘信息和团队待办，真正变成可处理、可追踪、可审批、可审计、可复盘的业务流程。

---

## 一、产品为什么存在

很多企业每天的大量工作都发生在聊天软件里：

- 客户在群里反馈系统故障，但没有自动形成工单。
- 潜在客户询问价格，销售没有及时记录和跟进。
- 社群里出现投诉、高意向用户或反复提问，但运营人员很难持续巡视。
- 候选人简历、岗位沟通和入职事项散落在聊天记录中。
- AI 能生成回答，但企业不敢让它未经审核直接对外发送。
- 管理者只能靠人工日报了解进度，无法追踪信息是如何被处理的。

WorkBuddy OSS 的核心思路是：

> 消息不是聊天记录的终点，而应该成为企业工作流的起点。

系统将每一条重要消息转化为结构化事件，再根据业务意图交给对应的场景 Agent，生成业务对象、处理建议和审批草稿，最终形成团队可以持续管理的业务闭环。

---

## 二、产品不是什么

为了正确理解产品，需要先明确它的边界。

WorkBuddy OSS：

- 不是一个单纯回答问题的聊天机器人。
- 不是 Dify、Coze、RAGFlow 一类通用 AI 应用搭建平台。
- 不是完整替代 CRM、ATS、SCRM 或大型 OA 的重型系统。
- 不负责训练和托管大模型。
- 不鼓励 AI 绕过人类直接对外承诺价格、合同、退款、Offer 等高风险内容。

它位于聊天工具和企业业务系统之间，负责把消息变成可以落地执行的工作。

---

## 三、产品最终目的

WorkBuddy OSS 的最终目的可以概括为五句话：

1. 让重要消息不再遗漏。
2. 让聊天内容自动变成业务对象和待办。
3. 让 AI 提高效率，但不越过人的决策权。
4. 让每一次处理、审批和发送都有记录。
5. 让企业能够在自己的服务器中安全运行整套系统。

它希望帮助企业建立下面这条标准链路：

```text
消息进入
  -> 标准化与去重
  -> AI/规则识别意图与风险
  -> 生成工单、线索、任务、候选人或知识缺口
  -> AI 生成处理建议和回复草稿
  -> 人工审批、编辑或拒绝
  -> 执行发送或内部处理
  -> 形成审计、报告和知识沉淀
```

## 四、完整产品架构

### 1. 接入层

负责接收企业已有的信息来源：

- 飞书真实消息和互动卡片。
- 企业微信加密回调和消息发送。
- 钉钉 Adapter 测试与扩展边界。
- 通用 Webhook。
- CSV、JSON 和文本聊天记录。
- Markdown、FAQ 和 CSV 知识材料。

### 2. 消息标准化层

不同平台的消息格式完全不同。WorkBuddy 会把它们统一转换为 `MessageEvent`，并记录：

- 消息来源和渠道。
- 会话和发送人。
- 文本、图片、文件或富文本类型。
- 原始消息内容。
- 平台消息 ID。
- 北京时间。
- 去重与幂等结果。

### 3. Agent Router

Agent Router 会结合规则、消息上下文和模型判断：

- 这是什么业务意图。
- 应交给哪个场景 Agent。
- 风险等级是什么。
- 置信度是多少。
- 是否需要人工介入。

确定性较高的内容优先走规则，复杂内容再交给模型，降低成本并提高可解释性。

### 4. 业务对象层

消息会进一步转化为：

- `Ticket`：客服工单。
- `Lead`：销售线索。
- `Task`：跟进任务。
- `Candidate`：候选人。
- `KnowledgeGap`：知识缺口。
- `KnowledgeItem`：正式知识条目。
- `Report`：业务报告。
- `Approval`：待人工确认的对外动作。

### 5. 人机协同层

AI 负责：

- 提取信息。
- 分类和评分。
- 生成建议。
- 生成回复草稿。
- 推荐下一步。

人负责：

- 审批。
- 编辑。
- 拒绝。
- 分配任务。
- 推进业务阶段。
- 决定是否真实对外发送。

### 6. 治理与运维层

系统还提供：

- 正式账号登录。
- 管理员、审批人、处理人和只读角色。
- 敏感配置加密。
- 操作审计。
- Agent 运行日志。
- 连接器诊断。
- 后台任务。
- 备份与恢复演练。
- Docker、PostgreSQL、Redis 和 HTTPS 私有部署。

## 五、推荐的产品讲解顺序

给用户演示时，不建议按照左侧菜单机械介绍。推荐按照真实工作发生的顺序讲：

1. 产品定位和企业痛点。
2. 登录、团队角色和安全边界。
3. 配置模型与消息渠道。
4. 消息如何进入系统。
5. 消息如何被标准化和路由。
6. 如何生成不同业务对象。
7. 四大场景 Agent 如何工作。
8. AI 草稿如何进入审批。
9. 人工如何审批、编辑和发送。
10. 任务、知识和报告如何持续沉淀。
11. 审计、日志和诊断如何保障运行。
12. 企业如何私有化部署。

这条讲解线路能够让用户看到一个完整故事，而不是一组互不相关的功能页面。

## 六、各功能的使用方法、作用与目的

## 1. 首次初始化与正式登录

### 怎么使用

首次打开系统时，管理员设置：

- 用户名。
- 显示名。
- 至少 8 位密码。

初始化完成后，所有成员必须使用正式账号登录。

### 能实现什么

- 阻止未登录用户访问企业业务数据。
- 每个处理动作都能关联到真实登录账号。
- 支持修改密码、退出和会话过期重新登录。

### 目的

让系统从个人演示工具升级为团队可以共同使用的正式工作台。

## 2. 团队成员与权限管理

### 怎么使用

管理员在“团队成员”中创建账号并分配角色：

- 管理员：管理用户、配置和系统策略。
- 审批人：审核 AI 草稿和高风险动作。
- 处理人：处理工单、任务和业务对象。
- 只读：查看数据但不执行写操作。

管理员还可以重置密码、停用账号或调整角色。

### 能实现什么

- 不同岗位看到和执行不同操作。
- 防止普通处理人员修改渠道密钥或全局策略。
- 审批、发送和处理记录归属具体人员。

### 目的

建立最基本的企业权限边界和责任边界。

## 3. 配置中心

### 怎么使用

管理员可在页面中配置：

- 模型服务商：Mock、DeepSeek、OpenAI 兼容服务、通义千问、Moonshot 等。
- Base URL、模型名、API Key 和超时时间。
- 飞书 App ID、Secret、审批群和校验参数。
- 企业微信 Corp ID、Agent ID、Secret、Token 和 EncodingAESKey。
- 钉钉 Client ID、Secret 和机器人参数。
- 是否启用真实 IM Adapter。
- 是否允许真实外发。
- 默认模拟发送或真实发送策略。

保存后可执行模型 Smoke Test 和渠道诊断。

### 能实现什么

- 不修改代码即可切换模型与渠道。
- 密钥只显示“已配置”，不会回显真实值。
- 敏感信息进入本地加密仓库。
- 支持旧明文密钥迁移和主密钥轮换。
- 配置在 Docker 容器重建后仍可保留。

### 目的

让企业管理员可以安全、可视化地完成系统配置，不需要开发人员直接修改源码。

## 4. 消息导入

### 怎么使用

“消息导入”支持：

- 上传 JSON。
- 上传 CSV。
- 粘贴文本。
- 预览解析结果。
- 确认后导入。

同一页面也支持导入 Markdown、FAQ 文本和 CSV 知识材料。

### 能实现什么

- 导入历史聊天记录做业务分析。
- 在没有真实 IM 接入前完成产品试用。
- 将旧客服记录、销售沟通和 FAQ 迁入系统。
- 在导入前发现格式错误和异常行。

### 目的

降低企业试用门槛，并给历史数据一个结构化入口。

## 5. 渠道接入与消息接收

### 怎么使用

- 飞书本地接收可启动长连接 Worker。
- 飞书卡片按钮使用公网 HTTPS 回调。
- 企业微信使用公网 HTTPS 加密回调。
- Webhook 可由其他系统推送消息。
- Adapter 测试台可先使用样例消息验证解析。

### 能实现什么

- 接收真实群聊和私聊消息。
- 识别文本、图片、文件和富文本。
- 映射发送人和会话。
- 同一平台消息 ID 重试时不会重复入库。
- 保存原始事件，出现问题时可追查或重放。

### 目的

把企业分散的消息入口统一到同一条业务处理链路中。

## 6. 消息事件

### 怎么使用

在“消息事件”中可以：

- 按来源、类型、Agent、风险和业务对象筛选。
- 搜索消息、发送人、会话和 Agent。
- 查看原始消息和标准化结果。
- 查看意图、风险、置信度和目标 Agent。
- 查看关联的工单、线索、任务或候选人。
- 查看对应 AgentRun。
- 在需要时重新执行路由。

### 能实现什么

让用户清楚回答：

- 这条消息从哪里来。
- 系统如何理解它。
- 它被交给了哪个 Agent。
- 它产生了什么业务结果。
- 整条链路是否完整。

### 目的

让 AI 的判断过程可以被检查，而不是成为黑盒。

## 7. 工作台

### 怎么使用

登录后首先查看工作台，其中包括：

- 消息、审批、工单、线索、任务、候选人、知识缺口和报告数量。
- 我的待办。
- 我的逾期。
- 我的审批。
- 待认领任务。
- 四大 Agent 总览。
- 风险聚合。
- 最近审计动作。
- 飞书连接状态。
- 正式工作流闭环和运营概览。

### 能实现什么

成员可以快速找到自己当天需要处理的事情，管理者可以快速看到团队风险和业务积压。

### 目的

把分散页面中的关键信息集中成日常工作的第一入口。

## 8. 业务对象中心

### 怎么使用

业务对象中心统一显示：

- 工单。
- 线索。
- 任务。
- 候选人。
- 知识缺口。
- 知识条目。
- 报告。
- 待审批事项。

点击对象可以进入对应场景或打开详情抽屉，查看来源消息、Agent、审批、处理记录和时间线。

### 能实现什么

把“消息视角”切换为“业务视角”，帮助用户围绕事项而不是围绕聊天窗口工作。

### 目的

建立 WorkBuddy OSS 最核心的业务对象中台。

## 9. 客服工单与知识 Agent

### 怎么使用

当消息包含登录失败、故障、投诉、退款或产品问题时，系统可生成客服工单。客服人员可以：

- 查看客户、优先级和 SLA。
- 将工单推进为待处理、处理中、等待客户、已解决或已关闭。
- 调整优先级。
- 查看来源消息。
- 查看完整业务时间线。
- 检索相关知识。
- 将未知问题沉淀为知识缺口。
- 生成回复草稿并送入审批。

### 能实现什么

- 防止客户问题停留在群消息中无人处理。
- 自动识别高风险或超时工单。
- 将解决方案反向沉淀到知识库。
- 让后续相同问题处理得更快。

### 目的

形成“客户问题 → 工单处理 → 知识复用 → 回复审批”的客服闭环。

## 10. 销售线索 Agent

### 怎么使用

当消息包含价格、方案、采购、合作或产品兴趣时，系统可生成销售线索。销售人员可以：

- 查看客户、公司、兴趣点和评分。
- 查看高意向、今日应跟进、停滞线索和谈判中线索。
- 将阶段推进为新线索、已联系、已发方案、谈判、赢单或输单。
- 查看下一步建议。
- 生成销售沟通草稿。
- 建立关联跟进任务。

### 能实现什么

- 从聊天记录中自动发现商机。
- 减少销售人员手工填写 CRM 的负担。
- 及时发现高意向和长期未跟进客户。
- 保留销售推进过程。

### 目的

形成“询价/合作消息 → 线索识别 → 销售跟进 → 阶段推进 → 成交复盘”的轻量销售闭环。

## 11. 私域社群运营 Agent

### 怎么使用

社群运营页面提供：

- 社群会话活跃度。
- 高意向用户。
- 未回复问题。
- 风险消息。
- 社群任务。
- 群消息数量和风险概览。

运营人员可以查看风险详情、生成回复草稿、送入审批，并完成社群任务。

### 能实现什么

- 从大量群消息中识别高价值用户。
- 发现反复无人回答的问题。
- 捕捉投诉、负面情绪和风险内容。
- 将运营动作变成明确任务。
- 为群回复建立审批边界。

### 目的

让社群运营从人工巡视升级为持续监测和任务化管理。

## 12. 跟进任务

### 怎么使用

任务可以来自客服、销售、社群或招聘流程。用户可以：

- 查看状态、优先级、负责人和截止时间。
- 筛选自己的任务、逾期任务和待认领任务。
- 认领任务。
- 推进为处理中、等待中、已完成或已取消。
- 查看关联业务对象和来源。
- 添加处理记录。

### 能实现什么

- 把 Agent 的建议变成真实待办。
- 明确负责人和截止时间。
- 发现逾期和无人认领事项。
- 保留执行过程。

### 目的

确保系统不仅“发现问题”，还推动问题被真正处理。

## 13. 招聘与入职 Agent

### 怎么使用

候选人页面可以：

- 查看候选人、岗位、阶段和匹配分。
- 推进筛选、面试、Offer、入职、已入职或淘汰。
- 查看多维匹配分析。
- 查看候选人优势、风险和能力缺口。
- 自动生成结构化面试问题。
- 管理入职 Checklist。
- 查看来源消息和完整时间线。

### 能实现什么

- 从简历或招聘沟通中提取候选人信息。
- 辅助 HR 与业务面试官进行初步匹配。
- 为面试提供有目的的问题清单。
- 防止入职事项遗漏。
- 对 Offer、录用和拒绝等高风险内容建立审批。

### 目的

形成“候选人信息 → 岗位匹配 → 面试 → Offer → 入职准备”的轻量招聘闭环。

## 14. 知识库与 RAG

### 怎么使用

知识库包括：

- 知识缺口。
- 正式知识条目。
- 知识图谱。
- 质量看板。
- 混合检索。
- 命中反馈。
- 版本历史和回滚。
- Obsidian 导出草案。

用户可以采纳知识缺口、编辑答案、发布或归档条目、设置复审日期、重建向量索引，并评价检索结果是否有帮助。

### 能实现什么

- 自动发现团队经常遇到但没有标准答案的问题。
- 将处理经验变成可搜索知识。
- 使用关键词、向量相似度和质量分进行混合检索。
- 为 AI 草稿提供带来源的知识引用。
- 低质量知识自动进入复审。
- 错误修改可以回滚。

### 目的

建立“业务处理产生知识，知识反过来提升业务处理”的持续学习闭环。

## 15. 审批队列

### 怎么使用

审批队列支持：

- 按状态、Agent 和业务对象筛选。
- 查看原始消息和 AI 草稿。
- 查看风险、意图、知识引用和业务上下文。
- 直接通过。
- 编辑后通过。
- 拒绝并填写原因。
- 查看发送预检。
- 模拟发送或真实发送。
- 查看发送尝试、错误和重试策略。
- 发送飞书互动审批卡片。

在飞书卡片中审批后，卡片会更新为“已通过”或“已拒绝”，并保留“查看详情”。

### 能实现什么

- 阻止 AI 未经确认直接对客户作出承诺。
- 让审批人可以修改措辞。
- 为高风险内容保留人工决策。
- 记录每一次发送及失败重试。

### 目的

这是 WorkBuddy OSS 的核心安全设计：AI 提效，人掌握最终决定权。

## 16. 报告中心

### 怎么使用

报告中心支持生成和查看：

- 业务运营报告。
- 客服日报。
- 销售日报。
- 社群日报。
- 招聘进度。
- 知识缺口报告。

报告会汇总关联消息、对象、风险和处理情况。

### 能实现什么

- 自动将大量消息和业务对象转为管理摘要。
- 减少人工整理日报和周报。
- 帮助管理者发现积压、风险和趋势。

### 目的

让企业从单条消息处理上升到整体运营管理和复盘。

## 17. 操作审计与 Agent 运行日志

### 怎么使用

“操作审计”用于查看：

- 谁在什么时间执行了什么动作。
- 动作所属范围。
- 动作状态。
- 关联业务对象。
- 操作摘要。

“运行日志”用于查看：

- 哪个 Agent 被调用。
- 输入消息。
- 模型和 Prompt 版本。
- 意图、风险和置信度。
- 输出和结构化动作。
- 耗时、错误和审批关系。

### 能实现什么

- 追溯成员操作。
- 解释 AI 为什么产生某个结果。
- 定位失败环节。
- 支持内部审计和责任确认。

### 目的

让企业自动化过程可解释、可追踪、可复盘。

## 18. 渠道会话、渠道事件与诊断

### 怎么使用

系统提供：

- 渠道会话：查看不同会话和发送策略。
- 渠道事件：查看接收成功、失败、重试和原始事件。
- Adapter 测试台：用样例验证飞书、企微和钉钉消息解析。
- 飞书诊断：检查 Worker、Token、卡片回调、发送条件和最近链路。
- 企微诊断：检查回调、验签解密、发送条件和最近链路。

### 能实现什么

- 在不查看服务器代码的情况下定位渠道问题。
- 区分消息长连接、HTTP 回调和卡片回调。
- 对失败事件进行重试。
- 在真实外发前完成安全复验。

### 目的

降低企业接入和长期运维消息平台的难度。

## 19. 演示模式

### 怎么使用

在“演示模式”中一键准备安全演示数据，然后按照页面给出的步骤依次体验消息、对象、工单、线索、社群、招聘、知识、报告和审批。

### 能实现什么

- 不接真实平台即可理解产品。
- 不发送真实消息。
- 快速完成售前演示、培训和功能验收。

### 目的

降低首次体验成本，同时避免误触真实外发。

## 七、一个完整使用案例

假设客户在飞书群里发送：

> 系统登录不上，一直报错，客户很着急。

WorkBuddy OSS 的处理过程如下：

1. 飞书 Worker 接收消息。
2. 系统根据平台消息 ID 去重。
3. 消息被转换为统一的 MessageEvent。
4. Agent Router 判断这是客服故障场景，风险较高。
5. 客服工单 Agent 生成一条高优先级 Ticket。
6. 系统检索知识库，寻找登录故障相关解决方案。
7. AI 生成回复草稿并附带知识引用。
8. 草稿进入审批队列。
9. 审批人在 WorkBuddy 或飞书卡片中查看上下文。
10. 审批人可以通过、编辑后通过或拒绝。
11. 通过后，系统根据会话策略模拟或真实发送。
12. 发送结果、平台消息 ID、操作人和时间写入审计。
13. 工单继续由处理人推进，必要时生成跟进任务。
14. 如果知识不足，系统产生 KnowledgeGap。
15. 问题解决后，经验沉淀为 KnowledgeItem。
16. 客服日报汇总本次工单和处理结果。

这就是产品的完整价值：一条聊天消息最终变成了可管理、可执行、可复盘的企业工作。

## 八、企业如何部署和开始使用

## 方案一：本地单机试用

适合个人体验、产品演示和小范围验证。

```bash
git clone <仓库地址>
cd "WorkBuddy OSS"

python3 -m venv .venv
.venv/bin/pip install -r apps/api/requirements.txt
npm --prefix apps/web install

npm run db:migrate
npm run services:start
```

浏览器打开：

```text
http://localhost:5173
```

首次进入时初始化管理员，然后使用演示模式或导入历史消息。

本地默认使用 SQLite，默认模型为 Mock，默认禁止真实外发。

## 方案二：企业私有化部署

适合正式团队运行。

推荐环境：

- Ubuntu LTS 云服务器。
- 2 核 4 GB 可用于小团队起步；消息量、并发和模型服务增大时应扩容。
- Docker 与 Docker Compose。
- PostgreSQL。
- Redis。
- Caddy HTTPS 反向代理。
- 独立域名。
- SSH 密钥登录。

生产栈包括：

```text
Caddy
  -> Web
  -> API
  -> PostgreSQL
  -> Redis
  -> runtime-jobs
  -> Feishu Worker
```

安全组只应开放：

- `22/tcp`
- `80/tcp`
- `443/tcp`

不要公开暴露：

- API `8000`
- Web 开发端口 `5173`
- PostgreSQL `5432`
- Redis `6379`

部署步骤：

1. 准备 Ubuntu 服务器和 SSH 密钥。
2. 配置域名解析。
3. 运行服务器初始化脚本。
4. 复制 `.env.production` 示例并设置域名和数据库密码。
5. 保持 `ENABLE_EXTERNAL_SEND=false`。
6. 执行部署脚本。
7. 检查 HTTPS 首页和 `/health`。
8. 初始化管理员账号。
9. 在配置中心填写模型与渠道参数。
10. 先完成模拟发送验收。
11. 再接入飞书或企微真实消息。
12. 最后在测试会话中受控开启真实外发。

## 正式启用建议

企业不要一次性全量上线，建议分四步：

### 第一阶段：安全演示

- 使用 Mock 模型。
- 外部发送关闭。
- 使用演示数据。
- 了解完整业务链路。

### 第二阶段：历史数据验证

- 导入 CSV/JSON 聊天记录。
- 检查路由和业务对象质量。
- 调整模型和规则。

### 第三阶段：真实接收、模拟发送

- 接入测试飞书或企微。
- 接收真实消息。
- AI 草稿进入审批。
- 所有发送保持模拟。

### 第四阶段：受控真实运行

- 只在测试群和测试用户中开启真实发送。
- 完成审批、审计、备份和恢复验证。
- 再逐步扩大到真实业务团队。
