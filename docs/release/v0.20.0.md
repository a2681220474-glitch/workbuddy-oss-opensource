# WorkBuddy OSS v0.20.0 Release Candidate Start

`v0.20.0` starts the release-candidate and open-source preparation line after
the `v0.19.x` knowledge system work.

## What Changed

### Added

- Added `docs/INSTALL.md` as the install entry for local release-candidate use.
- Added `docs/RELEASE_MODES.md` to separate demo mode, real-connector test mode,
  and empty-environment startup.
- Added `scripts/check_release_readiness.py`.
- Added `npm run check:release`.

### Optimized

- Version metadata is now synchronized across root package, web package,
  web lockfile, API `/health`, and the topbar.
- The release readiness check verifies required docs, version sync, safe
  `.env.example` secret defaults, gitignore secret rules, license, and
  contribution guide presence.

### Fixed

- Release-candidate setup is no longer scattered only across README,
  Quickstart, Runtime Config, and continuation notes.
- Web package lockfile version metadata no longer lags behind the package
  version.

## Final Result

WorkBuddy now has a clear release-candidate starting point:

```text
INSTALL -> RELEASE_MODES -> check:release -> normal build/compile/migrate checks
```

## Manual Acceptance

Run:

```bash
npm run build:web
.venv/bin/python -m compileall apps/api
npm run db:migrate
npm run check:release
```

Then verify:

1. `http://127.0.0.1:8000/health` returns version `0.20.0`.
2. `docs/INSTALL.md` explains local setup from clone to first run.
3. `docs/RELEASE_MODES.md` makes demo mode and real-connector mode distinct.
4. `npm run check:release` passes.
