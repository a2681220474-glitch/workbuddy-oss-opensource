# WorkBuddy OSS v1.1.7 本地正式收口

`v1.1.7` 在 `v1.1.6` 综合差距审计基础上，把“本地正式收口”变成可展示、可执行、可交接的发布边界。

## 本版做了什么

- 新增正式收口审计结果 `formal_closure`。
- 新增 `npm run check:formal-closure`。
- 新增聚合发布检查命令 `npm run check:formal-release`。
- 在配置中心展示维护模式边界。
- 更新 release、continuation、RC checklist 和 公开项目文档。

## 新增能力

- 配置中心可直接看到本地正式收口状态。
- 可直接看到聚合检查命令。
- 可直接看到允许维护、冻结范围和需明确授权事项。
- 发布前不再依赖人工记忆一串命令。
- 隔离验收会验证收口边界不会误触真实外发或远程部署。

## 优化

- 将“本地代码已完成”和“正式私有化已可投入”继续分开表达。
- 将维护模式边界从文档口径同步到 API 与前端。
- 将发布前主检查聚合成一个稳定入口。
- 将真实连接器复验、Docker/Postgres/Redis、ECS 升级继续保留为授权项。

## 修复

- 修复收口阶段仍只能靠文字交接判断的问题。
- 修复发布前检查命令分散在多个文档中的问题。
- 修复维护模式边界缺少机器可读输出的问题。
- 修复连接器复验和远程部署容易被误认为自动进入下一步的问题。

## 最终结果

- 本地正式收口状态：已就绪。
- 本地代码缺口：0。
- 聚合发布检查入口：`npm run check:formal-release`。
- 允许维护：P0/P1、权限/密钥/审计、安全修复、文档与验收一致性。
- 冻结范围：新增大业务模块、新渠道主线、大范围权限/部署重做、未授权真实外发、未授权 ECS 升级。
- 远程 ECS 继续冻结在 `v1.0.1`，本版未部署远程，也未触发真实外发。
- 隔离 Chrome 验收确认正式收口卡片显示完整且页面无横向溢出。

## 人工验收

1. 打开 `http://localhost:5173/#config`。
2. 找到“发布前综合差距审计”。
3. 确认出现“本地正式收口”。
4. 确认聚合检查命令为 `npm run check:formal-release`。
5. 确认允许维护、冻结范围、需明确授权三组标签存在。
6. 运行 `npm run check:formal-closure`，确认 10 项通过。
7. 运行 `npm run check:formal-release`，确认聚合检查通过。

## 自动验证

```bash
npm run check:formal-closure
npm run check:release-gap-audit
npm run check:formal-release
npm run check:rc
npm run check:local-acceptance
npm run build:web
npm run check:web-bundle
.venv/bin/python -m compileall apps/api scripts
```

隔离验收使用临时数据库、临时认证密钥和禁止外发策略完成，不修改正式业务数据。

## 下个版本规划

`v1.1.8` 进入本地维护稳定性扫尾：只处理收口后发现的文档一致性、检查命令可靠性、控制台警告和轻量体验缺口；继续保持远程 ECS 冻结。真实连接器外发和远程部署只在用户明确授权后执行。
