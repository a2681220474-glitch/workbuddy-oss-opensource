# WorkBuddy OSS v0.18.2 Migration And Runtime Jobs

`v0.18.2` continues the `v0.18.x` major line by turning two deployment goals
into real executable capabilities:

- schema migration baseline
- first background-job worker

## What Landed

- Added Alembic baseline:
  - `alembic.ini`
  - `alembic/env.py`
  - `alembic/versions/20260604_0001_baseline.py`
- Added migration runner:
  - `scripts/run_migrations.py`
  - `npm run db:migrate`
- Updated API startup rules:
  - SQLite local mode still bootstraps automatically
  - non-SQLite databases now require schema migration first
- Added runtime background-jobs worker:
  - `apps/api/workers/runtime_jobs.py`
  - `npm run dev:runtime-jobs`
  - `npm run check:background-jobs`
- Added runtime background status tracking:
  - worker heartbeat
  - latest cycle summary
  - latest worker error
- Added first real background-job cycle:
  - failed approval delivery retry scan
  - overdue follow-up task scan
  - stale open ticket scan
- Updated Compose stack:
  - API runs migrations before startup
  - Feishu worker runs migrations before startup
  - new `runtime-jobs` service

## Verification

- `.venv/bin/pip install -r apps/api/requirements.txt`
- `npm run db:migrate`
- `npm run check:background-jobs`
- `ENABLE_BACKGROUND_JOBS=true .venv/bin/python -m apps.api.workers.runtime_jobs --once`
- `npm run check:runtime-stack`
- `npm run build:web`
- `.venv/bin/python -m compileall apps/api`

## Manual Acceptance

1. Open `http://127.0.0.1:5173/#config`.
2. Confirm the Config Center now shows background-job worker state and local
   runtime-jobs start command.
3. Run `npm run db:migrate`.
4. Run `npm run check:background-jobs`.
5. Run `ENABLE_BACKGROUND_JOBS=true .venv/bin/python -m apps.api.workers.runtime_jobs --once`.
6. Open `http://127.0.0.1:8000/health`.
7. Confirm `version=0.18.2`, `background_jobs.queue_driver=database_polling`,
   and the worker cycle summary is present after the `--once` run.

## Notes

- The first background-job worker is intentionally small and deterministic. It
  does not yet do scheduled report creation or a full queue topology.
- Redis remains in the Compose target stack, but the first executable worker
  uses `database_polling` so we can start automating real work before full
  queue infrastructure is finished.
- Docker is not available in the current local runtime, so Compose parsing and
  full stack boot still need follow-up verification in a Docker-capable host.
