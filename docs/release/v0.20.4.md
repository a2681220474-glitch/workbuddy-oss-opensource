# WorkBuddy OSS v0.20.4 Release Candidate Closeout

`v0.20.4` closes the `v0.20.x` release-candidate and open-source preparation
line.

## What Changed

### Added

- Added `docs/RELEASE_CANDIDATE_CHECKLIST.md`.
- Added `docs/KNOWN_ISSUES.md`.
- Added `scripts/check_release_candidate.py`.
- Added `npm run check:rc`.
- Extended `npm run check:release` to require RC checklist and known issues
  docs.

### Optimized

- The RC checklist now ties together build, compile, migration, release,
  deployment, connector, knowledge, security, and manual product acceptance.
- Known issues now separate deployment, identity, AI/knowledge, connectors, and
  frontend limitations.
- The next major focus is explicitly framed as private-deployment v1.0.0
  readiness.

### Fixed

- Release-candidate acceptance no longer depends on scattered release notes and
  handoff memory.
- Unsupported claims are explicitly documented so the RC does not overstate
  Docker, RBAC, secret storage, or vector RAG readiness.

## Final Result

The `v0.20.x` line now has a single closeout path:

```text
release docs
  -> install/config/deploy/security docs
  -> known issues
  -> check:release
  -> check:deployment
  -> check:rc
  -> v1.0.0 readiness planning
```

## Manual Acceptance

Run:

```bash
npm run build:web
.venv/bin/python -m compileall apps/api scripts
npm run db:migrate
npm run check:release
npm run check:deployment
npm run check:rc
```

Then follow `docs/RELEASE_CANDIDATE_CHECKLIST.md`.

## Next Version

The next major line is `v1.0.0 私有化正式可用版`: production-grade auth/RBAC,
deployment verification, encrypted secret handling, stronger acceptance
automation, and final private-deployment readiness.
