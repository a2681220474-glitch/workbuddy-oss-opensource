# WorkBuddy OSS v0.18.3 Backup And Deployment Readiness

`v0.18.3` continues the `v0.18.x` deployment and runtime major line. This
version focuses on data safety and deployment verification before we keep
expanding scheduled work.

## What Landed

- Added backup management:
  - `scripts/manage_backups.py`
  - `npm run backup:create`
  - `npm run backup:verify`
  - `npm run backup:restore-plan`
  - `npm run backup:restore:sqlite`
- SQLite backups are created with SQLite's online backup API and verified with
  `PRAGMA integrity_check`.
- Backup metadata is written next to the backup artifact.
- PostgreSQL backup flow now has a `pg_dump` path and restore-plan guidance.
- Added deployment readiness check:
  - `scripts/check_deployment.py`
  - `npm run check:deployment`
- `/health` now includes backup runtime status.
- Config Center now shows:
  - backup readiness
  - backup directory
  - latest backup artifact
  - backup creation command
  - restore-plan command
- Version labels are updated to `v0.18.3`.

## Manual Acceptance

1. Open `http://127.0.0.1:5173/#config`.
2. Confirm the deployment card shows backup status, latest backup, and restore
   commands.
3. Run `npm run backup:create`.
4. Run `npm run backup:verify -- <backup-path>` using the generated backup path.
5. Run `npm run backup:restore-plan -- <backup-path>`.
6. Run `npm run check:deployment`.
7. Open `http://127.0.0.1:8000/health` and confirm `version=0.18.3` and a
   `backup` section are present.

## Notes

- SQLite restore remains explicit and requires `--confirm`.
- Backup artifacts live under `apps/api/data/backups` and should not be
  committed.
- Docker is still unavailable on the current local machine, so Compose full
  stack boot remains a follow-up item for a Docker-capable environment.

## Next Suggestion

Stay on `v0.18.x` and continue with structured logs, runtime error runbooks,
scheduled report generation, and Docker Compose full-stack validation.
