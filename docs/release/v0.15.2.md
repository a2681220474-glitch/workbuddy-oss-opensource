# WorkBuddy OSS v0.15.2 Feishu Failure Recovery

v0.15.2 continues the v0.15.x Feishu production loop. This version focuses on
failure recovery instead of new demo features: failed Feishu receive events can
be inspected and retried from the product, and approval sending now has a clear
retry policy with cooldown and maximum attempts.

## What Changed

- Added `POST /api/channel-events/{event_id}/retry` for retrying failed Feishu
  receive events that still contain a replayable raw payload.
- Marked retried ChannelEvents as `retried` and recorded retry activity in
  `AgentRun` with `agent_type=feishu_receive_retry`.
- Added retry metadata to ChannelEvent API responses, including retryability,
  attempt count, next attempt, and reason.
- Added a retry action and retry details to the Channel Events page.
- Added approval delivery retry policy:
  - maximum 3 delivery attempts
  - 60 second cooldown after the first failed attempt
  - 300 second cooldown after the second failed attempt
  - backend enforcement in both send preview and actual send
- Added retry policy visibility in the approval send modal and approval detail
  drawer.
- Bumped local app/API/runtime version to `0.15.2`.

## Manual Acceptance

1. Start API and Web:
   - `.venv/bin/uvicorn apps.api.main:app --host 127.0.0.1 --port 8000`
   - `npm --prefix apps/web run dev -- --host 127.0.0.1 --port 5173`
2. Open `http://localhost:5173/#channel-events`.
3. Filter Feishu failed events. Confirm retryable failures show `重试接收` and
   non-replayable failures explain why they cannot be retried.
4. Trigger a retry on a replayable failed Feishu event. Confirm the original
   event becomes `retried`, a new message or channel event is created, and a
   `feishu_receive_retry` AgentRun is recorded.
5. Open `http://localhost:5173/#approvals`.
6. For an approval with failed delivery, open send preview. Confirm it shows
   attempts, next attempt, cooldown, or retry limit.
7. Confirm backend blocks sending during cooldown or after the retry limit, even
   if the frontend action is called directly.
