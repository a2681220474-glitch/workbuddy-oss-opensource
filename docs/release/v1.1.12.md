# WorkBuddy OSS v1.1.12 北京时间与审批卡片状态

`v1.1.12` 修复线上 PostgreSQL 时间少 8 小时、飞书审批完成后卡片按钮不更新，以及“查看详情”按钮报错 `200672` 的问题。

## 本版做了什么

- API 业务响应统一把时间序列化为带 `+08:00` 的北京时间。
- 对 PostgreSQL 返回的无时区时间按 UTC 解释，再转换为 `Asia/Shanghai`。
- SQLite 本地历史时间继续按北京时间解释，不批量修改现有数据库。
- 飞书审批回调成功后直接返回更新后的原卡片。
- 审批通过后动作区显示“已通过”和“查看详情”。
- 审批拒绝后动作区显示“已拒绝”和“查看详情”。
- “查看详情”改为公网 URL 按钮，直达 `/#approvals?id=<审批ID>`，不再触发卡片回调。
- 连接器验收新增飞书消息 ID 幂等检查。

## 优化

- 时间输出包含明确时区，不再依赖浏览器猜测无时区字符串。
- 已处理审批不会继续展示可重复点击的通过、拒绝按钮。
- 飞书消息内容相同但消息 ID 不同的真实多次发送仍分别保留，避免误删有效业务消息。
- 同一个飞书消息 ID 因平台重试再次送达时会被忽略。

## 修复

- 修复 ECS PostgreSQL 页面时间比北京时间少 8 小时。
- 修复审批卡片完成后仍显示通过、拒绝按钮。
- 修复飞书卡片“查看详情”返回 `code: 200672`。
- 修复连接器验收未明确覆盖消息重试幂等和卡片完成态的问题。

## 最终结果

- 当前版本：`v1.1.12`。
- 远程 ECS 已完成受控升级，健康检查返回 `version=1.1.12`。
- 线上历史记录无需改库即可按北京时间正确显示。
- 飞书审批卡片按钮状态会随审批结果更新。
- 查看详情可打开线上审批队列并定位对应审批。
- 飞书 Worker 已重新连接官方长连接；公网回调 URL 验证返回正常。
- 截图中的相同文本记录具有不同飞书 `om_...` 消息 ID，属于不同时间实际发送的消息，不是同一事件重复入库。
- 本地正式收口状态保持不变，真实外发仍需用户明确授权。
- 远程 ECS 后续不随每个小版本自动升级；本版因用户明确要求修复线上问题而受控部署。

## 人工验收

1. 打开线上“消息事件”，确认时间比修复前增加 8 小时并与飞书客户端北京时间一致。
2. 打开“审批队列”，确认创建时间同样为北京时间。
3. 在飞书发送一条全新的测试消息，确认 WorkBuddy 只新增一条记录。
4. 给该审批发送飞书卡片并点击“通过”。
5. 确认原卡片动作区变为“已通过”和“查看详情”。
6. 点击“查看详情”，确认浏览器打开线上 WorkBuddy 审批队列并定位该审批，不再出现 `200672`。
7. 再次点击已通过状态按钮应不可操作。
