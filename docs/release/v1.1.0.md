# WorkBuddy OSS v1.1.0 本地密钥加密存储

`v1.1.0` 完成九项后续基线中的第 1 项：将 DeepSeek、飞书、企业微信和钉钉敏感配置从 `.env.local` 明文保存升级为本地加密存储。

## 本版做了什么

- 新增 Fernet 认证加密仓库 `apps/api/data/runtime_secrets.json`。
- 新增独立主密钥文件 `apps/api/data/runtime_secret.key`。
- 两个文件均只允许本机当前用户读写，权限为 `0600`，并继续由 Git 和 Docker build context 忽略。
- 配置运行时优先读取加密仓库，同时兼容尚未迁移的旧环境变量。
- 新保存的敏感配置直接写入加密仓库，不再写回 `.env.local`。
- 新增旧明文配置迁移、主密钥轮换和安全状态检查。
- 配置中心新增“敏感配置加密存储”面板。
- 审计总账记录迁移和轮换动作，但不记录密钥值。

## 新增能力

- 管理员可在配置中心执行“迁移现有明文”。
- 管理员可在配置中心执行“轮换主密钥”。
- 状态接口展示密文条目数、明文残留数、主密钥权限和存储健康状态。
- 新增 `npm run check:secret-storage`，在临时目录验证迁移、加解密、轮换、权限和密文不泄露。

## 优化

- 普通运行参数仍保留在 `.env.local`，敏感值与普通配置分离。
- 密钥输入框继续保持“留空不修改”，不会意外清空已有值。
- 加密仓库和主密钥均使用原子写入，降低中断造成文件损坏的风险。
- 迁移后的兼容备份也会移除敏感行，避免明文从备份泄露。

## 修复

- 修复 DeepSeek、飞书、企微等密钥长期明文保存在 `.env.local` 的安全缺口。
- 修复旧配置中心提示仍宣称密钥保存在环境文件的问题。
- 修复管理员权限错误信息仍引用旧 `v0.16.x` 的问题。
- 修复未登录打开业务页面时并发 `401` 触发重复鉴权和提示风暴的问题。

## 最终结果

- 本机已迁移 5 项既有敏感配置，迁移后明文残留为 0。
- DeepSeek、飞书和企微配置在迁移后仍被运行时正确识别。
- 主密钥和密文仓库权限均为 `0600`。
- 远程 ECS 仍冻结在 `v1.0.1`，本版未部署远程。

## 人工验收

1. 打开 `http://localhost:5173/#config`。
2. 确认“敏感配置加密存储”显示“正常”“残留明文：无”。
3. 确认已加密条目数大于 0，主密钥权限显示“仅本机用户可读”。
4. 在大模型配置中保持 API Key 留空并点击“测试模型调用”，确认原 DeepSeek 配置仍可用。
5. 修改一个测试密钥后保存，确认页面只显示“已配置”，不回显值。
6. 点击“轮换主密钥”，确认完成后配置状态和模型调用仍正常。
7. 在审计日志中确认存在迁移或轮换记录，且详情没有密钥值。
