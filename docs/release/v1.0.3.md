# WorkBuddy OSS v1.0.3 Local Same-Origin API Fix

`v1.0.3` is a small but important local-development fix. It keeps the new formal
login flow from being derailed by browser-side requests to `:8000`, and makes
the local web app behave like the production deployment path.

## What Changed

### Added

- Added Vite dev and preview proxy support for `/api`.
- Added local dev compose alignment so the web container also uses same-origin
  `/api`.

### Optimized

- The web client now defaults to same-origin `/api` instead of hard-coding a
  direct `:8000` API base.
- Local browser requests now stay on the web origin and are forwarded by Vite,
  which avoids browser-side client blocking in this environment.

### Fixed

- Fixed the local login/bootstrap page falling back to a misleading normal
  login state when direct requests to `:8000` were blocked.
- Fixed the browser-side `Failed to fetch` / invalid URL behavior that appeared
  on first load.

## Final Result

The local app now opens on the correct first-run bootstrap screen when the
tenant has no password user yet.

The browser no longer has to call the backend directly on `:8000` during local
development; it uses same-origin `/api` and Vite forwards those requests to the
API.

`/health` should now report version `1.0.3`.

## Manual Acceptance

1. Open `http://localhost:5173/`.
2. Confirm the first-run bootstrap screen is shown.
3. Confirm no `Failed to fetch` toast appears on initial load.
4. Set the local admin password and log in.
5. Reload the page and confirm the session remains usable.
6. Confirm `http://127.0.0.1:8000/health` reports `version: 1.0.3`.

## Verification

- `npm run build:web`
- Browser reload on `http://localhost:5173/`
- `http://localhost:5173/` shows the bootstrap page when no password user exists
