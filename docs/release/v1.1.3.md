# WorkBuddy OSS v1.1.3 本地服务托管与故障恢复

`v1.1.3` 完成九项后续基线中的第 4 项：本地 API、Web、后台任务和渠道 worker 的进程托管、健康检查、日志轮转与故障恢复。

## 本版做了什么

- 新增 `scripts/manage_local_services.py` 本地服务管理器。
- 默认托管 API、Web 和 runtime-jobs。
- 飞书长连接 worker 必须显式选择，不会随默认命令自动启动。
- 每个服务拥有独立 supervisor PID、child PID、状态文件和输出日志。
- 子进程异常退出后自动重启，并限制短时间重启风暴。
- 停止 supervisor 时同步终止其子进程组，避免遗留后台进程。

## 新增能力

```bash
npm run services:start
npm run services:status
npm run services:restart
npm run services:stop
npm run services:logs
```

按需管理 worker：

```bash
npm run services:start -- workers
npm run services:start -- feishu-worker
npm run services:stop -- feishu-worker
```

## 优化

- API 和 Web 使用 HTTP 健康探测。
- runtime-jobs 和飞书 worker 使用现有状态文件与心跳判断。
- 运行栈检查优先读取托管 API 的真实健康状态，database_polling 模式不再被未配置 Redis 误判为降级。
- 服务 stdout/stderr 按大小轮转，默认单文件 5 MiB、保留 3 份备份。
- API、runtime-jobs、飞书 worker 的结构化 JSONL 日志也使用同样的轮转基线。
- 配置中心和运行状态中的恢复命令已切换到托管入口。

## 修复

- 解决关闭终端后本地服务直接掉线的问题。
- 解决 worker 异常退出后只能人工发现和重启的问题。
- 解决长期运行时 stdout 和 JSONL 日志无上限增长的问题。
- 避免默认启动命令误连接真实飞书长连接。

## 最终结果

- 隔离故障测试验证首次崩溃后可自动拉起第二个子进程。
- PID 状态、重启次数、退出码和健康状态可查询。
- 服务日志和结构化日志均完成轮转验证。
- 本地 API 与 Web 已切换到服务管理器运行。
- 远程 ECS 继续冻结在 `v1.0.1`，本版未部署远程。

## 人工验收

1. 运行 `npm run services:status`，确认 API、Web 和 runtime-jobs 为 `ok`。
2. 打开 `http://localhost:5173/`，确认顶部显示 `v1.1.3 服务托管`。
3. 运行 `npm run services:logs -- api web`，确认可以查看近期输出。
4. 运行 `npm run services:restart -- web`，刷新页面确认 Web 自动恢复。
5. 运行 `npm run check:local-services`，确认 9 项隔离验收全部通过。
6. 只有需要真实接收飞书消息时，才运行 `npm run services:start -- feishu-worker`。
