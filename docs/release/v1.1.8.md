# WorkBuddy OSS v1.1.8 维护稳定性扫尾

`v1.1.8` 在 `v1.1.7` 本地正式收口之后，专门处理浏览器验收发现的非阻塞控制台提示，并把前端维护项纳入正式发布检查。

## 本版做了什么

- 前端入口增加 Ant Design `App` 上下文 Provider。
- 登录页、应用外壳和配置中心改用上下文里的 `message`。
- 配置中心改用上下文里的 `modal.confirm`。
- 修改密码弹窗将 `destroyOnClose` 替换为 `destroyOnHidden`。
- 将 Ant Design CSS-in-JS 运行时拆为稳定 vendor chunk，保持入口 bundle 预算。
- 新增 `npm run check:frontend-maintenance`。
- 将前端维护检查并入 `npm run check:formal-release`。
- 更新 release、continuation、Known Issues 和 公开项目文档。

## 新增能力

- 发布前可以自动检查 Ant Design 上下文 Provider 是否存在。
- 发布前可以自动检查关键验收链路是否继续使用上下文 message/modal。
- 发布前可以自动阻止 `destroyOnClose` 弃用属性回流。
- 配置中心和正式收口文档继续保留“远程 ECS 冻结、真实外发需授权”的边界。

## 优化

- 清理登录、修改密码、配置中心主验收链路的控制台噪音。
- 减少人工浏览器验收时的误判成本。
- 入口 chunk 从约 505.6 kB 降回约 483.3 kB，继续低于 500 kB 预算。
- 将维护扫尾从口头事项变成可重复执行的脚本检查。
- 保持 v1.1.x 的本地优先节奏，不触碰远程 ECS。

## 修复

- 修复 Ant Design `message` 静态调用在动态主题上下文下的警告。
- 修复配置中心 `Modal.confirm` 静态调用的上下文警告风险。
- 修复 `Modal destroyOnClose` 弃用提示。
- 修复版本号与本地验收脚本仍停留在 `v1.1.7` 的问题。

## 最终结果

- 本地版本推进到 `v1.1.8`。
- 前端维护检查：`npm run check:frontend-maintenance`。
- 聚合发布检查：`npm run check:formal-release` 已包含前端维护检查。
- 远程 ECS 继续冻结在 `v1.0.1`，本版未部署远程。
- 本版未触发真实飞书、企微、钉钉外发。

## 人工验收

1. 打开 `http://localhost:5173/`。
2. 登录或完成首次初始化。
3. 确认工作台顶部版本显示 `v1.1.8 维护稳定性扫尾`。
4. 打开 `http://localhost:5173/#config`。
5. 打开浏览器控制台，确认没有 `[antd: message] Static function can not consume context`。
6. 确认没有 `destroyOnClose is deprecated`。
7. 运行 `npm run check:frontend-maintenance`，确认全部通过。
8. 运行 `npm run check:formal-release`，确认聚合发布检查通过。

## 自动验证

```bash
npm run check:frontend-maintenance
npm run check:formal-release
npm run check:rc
npm run check:local-acceptance
npm run build:web
npm run check:web-bundle
.venv/bin/python -m compileall apps/api scripts
```

隔离浏览器验收使用临时数据库、临时认证密钥和禁止外发策略完成，不修改正式业务数据。

`v1.1.9` 继续本地维护收口：优先处理深层页面交互中可能残留的静态 Ant Design 调用、轻量 UI 自动化覆盖和文档一致性；继续保持远程 ECS 冻结。真实连接器外发和远程部署只在用户明确授权后执行。
