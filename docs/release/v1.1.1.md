# WorkBuddy OSS v1.1.1 隔离式产品全链路验收

`v1.1.1` 完成九项后续基线中的第 5 项：新增不接触真实业务数据的产品全链路自动化验收。

## 本版做了什么

- 新增 `scripts/check_product_workflow.py`。
- 新增命令 `npm run check:product-workflow`。
- 验收脚本使用临时 SQLite、临时认证密钥和临时加密配置仓库。
- 脚本运行前强制使用 Mock LLM、关闭真实 IM Adapter、外部发送和后台任务。
- 脚本结束后自动销毁全部验收数据和临时密钥。

## 新增能力

自动覆盖 20 个检查点：

1. API 版本和健康状态。
2. 首次管理员初始化。
3. 会话 Cookie 认证。
4. 错误密码拒绝。
5. 创建审批人和处理人。
6. 处理人越权修改配置被拒。
7. 临时 API Key 加密保存且响应不回显。
8. 加密仓库健康、无明文残留。
9. FAQ 知识导入并发布。
10. 知识检索和命中记录。
11. 消息导入和 Agent 路由。
12. 自动生成工单、审批和 AgentRun。
13. 消息与业务对象关联。
14. 业务对象中心统计。
15. 工单来源、Agent、审批和时间线。
16. 处理人新增处理记录。
17. 审批上下文包含知识引用。
18. 审批人审批并完成 Mock 发送。
19. 关键操作写入审计总账。
20. 数据库和密钥仅存在于临时目录。

## 优化

- 验收不要求本地 API 或 Web 事先启动。
- 不使用用户真实登录密码、真实 DeepSeek Key 或真实渠道密钥。
- 不修改 `apps/api/data/workbuddy.db`、`.env.local` 或现有加密仓库。
- 每个检查点单独输出通过或失败原因，方便定位回归位置。

## 修复

- 将“接口可访问”的浅层检查升级为包含业务结果断言的完整回归。
- 补齐认证、RBAC、知识引用和审计总账缺少自动联动验证的问题。
- 验证处理动作严格遵守正式枚举契约，错误动作会被 `422` 拒绝。

## 最终结果

- `npm run check:product-workflow` 共 20 项检查全部通过。
- 登录、用户、配置、知识、消息、工单、审批、Mock 发送和审计形成可重复回归链路。
- 本地真实数据和配置未被验收脚本修改。
- 远程 ECS 继续冻结在 `v1.0.1`，本版未部署远程。

## 人工验收

1. 在项目根目录运行 `npm run check:product-workflow`。
2. 确认最后显示 `Product workflow checks passed (20 checks).`
3. 运行前后打开现有本地工作台，确认历史消息、工单和配置没有增加验收数据。
4. 打开 `http://localhost:5173/`，确认顶部版本显示 `v1.1.1 自动化验收`。
5. 使用原账号登录，确认现有 DeepSeek、飞书和企微配置状态保持不变。

## 自动验证

```bash
npm run check:product-workflow
npm run check:secret-storage
npm run check:local-acceptance
npm run build:web
.venv/bin/python -m compileall apps/api scripts
```
