# WorkBuddy OSS v0.18.1 Deployment Runtime Foundation

`v0.18.1` is the first concrete delivery inside the `v0.18.x` major line.

It does not jump to a new product surface. Instead, it starts turning the
project from a developer-machine app into a runtime stack that can explain its
own deployment health.

## What Landed

- Added structured runtime-stack inspection in the API:
  - `GET /health`
  - database backend, connection state, persistence mode
  - Redis configuration and connectivity
  - background-job enablement and readiness
- Added shared runtime-status service used by health and Config Center.
- Added `npm run check:runtime-stack`.
- Added PostgreSQL runtime dependency:
  - `psycopg[binary]`
- Upgraded Docker Compose toward a fuller stack:
  - `postgres`
  - `redis`
  - `api`
  - `web`
  - `feishu-worker`
- Updated Config Center to display:
  - database connection state
  - Redis state
  - background-job readiness
  - Compose start commands
- Updated local runtime docs and environment template for Redis/background-job
  settings.

## Manual Acceptance

1. Open `http://127.0.0.1:5173/#config`.
2. Confirm the Config Center now shows:
   - database backend and connection state
   - Redis state
   - background-job state
   - Compose startup command
3. Open `http://127.0.0.1:8000/health`.
4. Confirm the health payload now includes `database`, `redis`,
   `background_jobs`, and `version=0.18.1`.
5. Run `npm run check:runtime-stack`.
6. Confirm the current local environment reports:
   - SQLite connected
   - Redis not configured
   - background jobs disabled

## Notes

- This machine is still in local SQLite mode. That is expected for the current
  manual runtime.
- Docker Compose now targets PostgreSQL + Redis, but full Compose validation
  still depends on having Docker available in the runtime environment.
- Background jobs are not yet executing real retry/report scans; this version
  lays down the queue and readiness contract first.
