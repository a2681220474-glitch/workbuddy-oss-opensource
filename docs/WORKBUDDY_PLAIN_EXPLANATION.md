# WorkBuddy OSS 通俗说明：这个项目到底是什么、怎么用、现在做到哪一步

版本：v0.14.6  
日期：2026-05-27  
说明对象：不写代码的老板、业务负责人、产品负责人、实施同事

---

## 1. 一句话说明

WorkBuddy OSS 不是一个普通聊天机器人，也不是 Dify、Coze 那种“搭 AI 应用的平台”。

它更像一个“IM 消息驱动的业务中台”：

> 把飞书、企业微信、钉钉、CSV、Webhook 里的聊天消息，自动整理成工单、销售线索、社群任务、候选人、知识缺口、日报和审批记录。

通俗一点说：

很多公司真实工作都发生在群聊里。客户问价格、用户投诉、销售跟进、HR 收简历、群里有人没人回复、同事说要处理一个问题。这些内容平时都散落在飞书、企微、钉钉里，靠人记，容易漏。

WorkBuddy OSS 想做的事情是：

1. 自动把聊天消息收进来。
2. 判断这条消息属于什么业务。
3. 自动生成对应的业务对象，比如工单、线索、任务、候选人、知识缺口。
4. 如果要对外回复客户，先生成草稿，放进审批队列。
5. 人确认后再发送，系统留下完整记录。
6. 最后沉淀成日报、知识库、审计日志，方便复盘。

---

## 2. 它不是做什么的

这个项目不是：

| 不是 | 为什么 |
|---|---|
| 不是 Dify / Coze 竞品 | 它不是让你拖拽搭通用 AI 应用 |
| 不是纯聊天机器人 | 目标不是“AI 陪聊”或“自动回一句话” |
| 不是完整 CRM / ATS / 客服系统 | 它做轻量业务闭环，不替代重型系统 |
| 不是模型平台 | 它不训练模型，只调用外部大模型 |
| 不是完全自动外发机器人 | 对外回复默认要进审批 |

它真正做的是：

> 把 IM 里的零散消息变成可跟踪、可审批、可复盘的业务流程。

## 3. 现在这些 Agent 是“正式可用”了吗？

要分两层看。

### 3.1 作为本地产品原型：已经能跑完整闭环

当前 v0.14.6 已经可以在本地跑通这些核心流程：

- 消息进入系统。
- 系统识别消息类型。
- 生成工单、线索、任务、候选人、知识缺口等业务对象。
- 生成审批草稿。
- 查看运行日志。
- 重跑消息路由。
- 重放某次 AgentRun。
- 查看前后对比。
- 配置真实大模型。
- 测试模型是否可调用。
- 飞书可以真实接收消息，前提是飞书 worker 正在运行。

所以如果你问：“这个项目是不是已经不是空壳了？”

答案是：是的，已经不是空壳，已经有可操作的本地产品闭环。

### 3.2 作为生产级商业系统：还不是最终正式版

如果你问：“现在能不能直接拿去给真实公司长期生产使用？”

答案是：还不能说是最终生产版。

原因是：

- 飞书已经能真实测试，但企微、钉钉还主要是配置入口和 mock/预留结构。
- 各 Agent 的业务逻辑还偏规则和模板，没有完全做到复杂场景下的稳定智能判断。
- 权限、多租户、部署、告警、数据备份、审计合规还没有做到企业生产级。
- 报告质量、知识库 RAG、附件解析、线索去重等能力还需要继续增强。

更准确的定位是：

> 当前是一个可本地验收、可演示、可继续迭代的业务 Agent 工作流产品原型，不是最终商业生产版。

## 4. 这些 Agent 到底是怎么工作的？是不是都在调用大模型？

当前不是每一步都调用大模型。

WorkBuddy 的 Agent 逻辑是“规则优先，大模型辅助”。

### 4.1 为什么不全部交给大模型？

因为企业业务系统里，很多事情需要稳定、可解释、可审计。

比如：

- 出现“退款”“投诉”“报错”，大概率就是客服工单。
- 出现“多少钱”“报价”“方案”，大概率就是销售线索。
- 出现“简历”“岗位”“JD”，大概率就是招聘场景。
- 出现“群里没人回复”“报名训练营”，大概率就是社群运营。

这些高确定性的场景，用规则更稳定，也更便宜。

### 4.2 当前调用大模型的地方

现在大模型主要用于：

1. 当规则判断不够明确时，辅助做意图分类。
2. 记录模型请求、模型输出、失败原因。
3. 未来可以继续扩展到更复杂的回复生成、报告总结、知识归纳。

当前配置中心已经可以配置 OpenAI-compatible 大模型：

- DeepSeek
- OpenAI-compatible
- OpenAI
- 通义/Qwen
- Moonshot

当前项目里的模型配置是：

- Provider：DeepSeek
- Model：deepseek-v4-flash
- Base URL：https://api.deepseek.com/v1

配置中心还提供“测试模型调用”按钮，用来确认模型是否真的能调用。

### 4.3 当前 Agent 的工作链路

一条消息进来后，大致是这样：

```text
飞书/CSV/JSON/Webhook 消息
  -> MessageEvent 标准消息
  -> Agent Router 判断属于哪个业务
  -> 对应场景 Agent 生成动作
  -> 创建业务对象
  -> 如需外发，进入审批队列
  -> 人工确认后发送
  -> 记录运行日志和审计信息
```

其中 Agent Router 会先看规则：

- 客服关键词 -> 客服工单 Agent
- 价格/报价/方案 -> 销售线索 Agent
- 群运营关键词 -> 社群运营 Agent
- 简历/JD/岗位 -> 招聘入职 Agent
- 判断不清楚 -> 调用大模型辅助分类
- 还是不确定 -> 进入人工收件箱

### 4.4 它不是“你手动点某个 Agent 聊天”

现在的 Agent 不是像 ChatGPT 那样你打开一个窗口和它聊天。

更准确地说：

> Agent 是后台业务处理模块。你给系统输入消息，系统自动决定该由哪个 Agent 处理。

你调用 Agent 的方式通常有三种：

1. 真实飞书消息进来，系统自动处理。
2. 在消息导入页导入 CSV/JSON/文本，系统自动处理。
3. 在消息事件页手动点“重跑”，让当前 Agent Router 重新判断。

还有一个高级能力：

4. 在运行日志页点“重放”，重放某次 AgentRun，并查看前后对比。

## 5. 当前有哪些 Agent？分别做什么？

## 5.1 客服工单与知识 Agent

适合处理：

- 客户投诉
- 系统报错
- 退款问题
- 产品使用问题
- 知识库没有答案的问题

它会做：

- 从消息生成 Ticket 工单。
- 标记优先级和状态。
- 生成待审批回复草稿。
- 记录 SLA 相关信息。
- 从工单沉淀 KnowledgeGap 或 KnowledgeItem。

例子：

用户在群里说：

> 系统登录不了，一直报错，客户很着急。

系统可能会：

- 创建一个客服工单。
- 标记风险/优先级。
- 生成一条待审批回复草稿。
- 后续可以沉淀成知识库问题。

当前状态：

- 已有工单列表、状态流转、SLA 配置、知识沉淀、客服日报。
- 还需要增强工单详情、处理记录、知识引用、审批草稿历史。

---

## 5.2 销售线索 Agent

适合处理：

- 报价咨询
- 方案咨询
- 试用咨询
- 客户表达购买意向
- 需要后续跟进的销售对话

它会做：

- 从消息生成 Lead 销售线索。
- 给线索打分。
- 生成下一步跟进建议。
- 创建跟进任务。
- 生成销售回复草稿并进入审批。

例子：

客户说：

> 我想了解一下你们多少钱，可以发报价方案吗？

系统可能会：

- 生成一条销售线索。
- 识别意向较高。
- 创建跟进任务。
- 生成报价回复草稿，等待审批。

当前状态：

- 已有线索列表、阶段流转、评分卡、跟进草稿、销售日报。
- 还缺线索去重/合并、报价模板、完整销售详情。

---

## 5.3 私域社群运营 Agent

适合处理：

- 群里有人问价格没人回复。
- 群里有高意向用户。
- 群里有人投诉、退款、负面反馈。
- 群里问题没人回答。
- 运营需要生成社群任务。

它会做：

- 识别高意向用户。
- 识别未回复问题。
- 识别风险消息。
- 生成社群运营任务。
- 给风险消息生成回复草稿，进入审批。

例子：

群里有人说：

> 群里有人问价格没人回复，可能是高意向用户。

系统可能会：

- 标记为社群运营场景。
- 记录风险消息。
- 生成社群任务。
- 支持人工生成回复草稿并送审批。

当前状态：

- v0.14.6 已加固社群页 UX。
- 多个 tab 表格支持横向滚动和列宽拖拽。
- 风险消息有详情抽屉。
- 社群任务有详情和完成处理。
- 企微真实群接入还没完成。

---

## 5.4 招聘入职 Agent

适合处理：

- 简历
- JD
- 岗位要求
- 候选人匹配
- 面试问题
- 入职 Checklist

它会做：

- 从消息生成 Candidate 候选人。
- 做岗位/JD/简历匹配。
- 生成面试问题。
- 生成入职 Checklist。
- 生成招聘进度报告。

例子：

HR 发来：

> 这个候选人的简历和销售岗位 JD 匹配度怎么样？

系统可能会：

- 创建候选人记录。
- 给匹配分。
- 生成面试问题。
- 生成后续入职检查项。

当前状态：

- 已有候选人模型、匹配分析、面试问题、入职 Checklist、招聘报告。
- 还缺真实附件简历解析、结构化 JD/简历解析增强。

---

## 5.5 知识库 Agent / 知识沉淀

适合处理：

- 客户问了知识库没有的问题。
- 工单中出现重复问题。
- 社群里有人问常见问题。
- 需要把答案沉淀成正式知识条目。

它会做：

- 创建 KnowledgeGap，表示“知识缺口”。
- 从工单或社群问题沉淀 KnowledgeItem。
- 支持采纳、忽略、发布、归档。
- 记录来源消息和引用示例。

当前状态：

- v0.14.5 已加固知识库页面。
- KnowledgeGap 和 KnowledgeItem 都有详情抽屉。
- 表格支持横向滚动和列宽拖拽。
- 还缺真正 RAG、引用来源增强、编辑发布体验加固。

## 6. 每个页面是干什么的？

## 6.1 业务工作台

老板或负责人看的总览页。

作用：

- 看系统里有多少消息。
- 有多少待审批。
- 有多少工单、线索、任务、候选人、知识缺口、报告。
- 快速判断系统是否在运转。

适合谁：

- 老板
- 运营负责人
- 项目负责人

---

## 6.2 消息导入

把历史聊天记录或测试数据导入系统。

支持：

- CSV
- JSON
- 文本
- Adapter payload

作用：

- 没接真实 IM 之前，可以先用历史聊天记录验证产品。
- 可以批量导入客户对话，让 Agent Router 生成业务对象。

---

## 6.3 消息事件

这是系统的“消息流水账”。

每条进入系统的消息都会在这里变成 MessageEvent。

你可以看：

- 消息时间
- 渠道
- 会话
- 发送人
- 消息内容
- 被识别成什么意图
- 交给哪个 Agent
- 风险等级
- 是否生成业务对象

重要操作：

- 查看运行详情。
- 重跑路由。

“重跑路由”的意思是：

> 用当前最新的 Agent Router 重新判断这条消息，并重新生成业务对象。

---

## 6.4 审批队列

所有对外回复草稿都会进这里。

为什么需要审批？

因为企业场景不能让 AI 随便对客户承诺价格、退款、合同、Offer 等敏感内容。

审批队列可以做：

- 查看 AI 生成的回复草稿。
- 编辑草稿。
- 批准。
- 拒绝。
- 预览发送。
- 真实发送或模拟发送。

当前飞书真实发送已经有能力，但仍建议测试时明确区分模拟和真实。

---

## 6.5 运行日志

这是系统的“黑匣子”。

每次 Agent Router 或适配器运行，都会记录 AgentRun。

可以看：

- 哪个 Agent 被调用。
- 为什么这么路由。
- 有没有调用大模型。
- 模型请求是什么。
- 模型输出是什么。
- 是否失败。
- 生成了什么动作。
- 耗时和 token。

重要操作：

- 查看详情。
- 重放运行。

“重放运行”的意思是：

> 找到某一次历史运行，用当前最新逻辑再跑一次，并显示前后对比。

这对调试 Agent 非常重要。

## 7. 正常使用顺序是什么？

如果从零开始，本地正常使用顺序建议是：

### 第一步：启动服务

启动 API：

```bash
.venv/bin/uvicorn apps.api.main:app --host 127.0.0.1 --port 8000
```

启动 Web：

```bash
npm --prefix apps/web run dev -- --host 127.0.0.1 --port 5173
```

打开：

```text
http://localhost:5173
```

### 第二步：进入配置中心

检查：

- API 是否正常。
- 大模型是否配置。
- 飞书是否配置。
- 发送模式是模拟还是真实。
- 飞书 worker 是否在线。

### 第三步：测试模型调用

在配置中心点击：

```text
测试模型调用
```

确认大模型可用。

### 第四步：导入消息或接收飞书消息

两种方式：

1. 用“消息导入”导入 CSV/JSON。
2. 启动飞书 worker，接收真实飞书消息。

飞书 worker 命令：

```bash
npm run dev:feishu-stream
```

### 第五步：看消息事件

进入“消息事件”：

- 看消息有没有进来。
- 看系统识别成什么意图。
- 看交给哪个 Agent。
- 看有没有生成业务对象。

### 第六步：看业务页面

根据消息类型去看：

- 客服工单与知识。
- 销售线索。
- 社群运营。
- 招聘入职。
- 知识库。

### 第七步：审批外发内容

如果系统生成了对外回复草稿，进入“审批队列”：

- 查看原消息。
- 查看 AI 草稿。
- 编辑。
- 批准。
- 发送。

### 第八步：看运行日志和报告

进入“运行日志”：

- 看每次 Agent 为什么这么判断。
- 看模型有没有失败。
- 看生成了什么动作。
- 必要时重放运行。

进入“报告中心”：

- 看日报和业务汇总。

## 8. 当前各渠道状态

## 8.1 飞书

当前飞书是最接近真实验收的渠道。

已经有：

- 飞书配置入口。
- 飞书真实消息接收能力。
- 飞书 worker 状态检查。

## 9. 这个项目对老板有什么价值？

## 9.1 防止重要消息漏掉

群里客户问价格、投诉、退款、报名、简历、问题没人回复，这些都容易被淹没。

WorkBuddy 可以把这些消息结构化成任务、工单、线索。

## 9.2 让业务过程可追踪

以前是：

> 群里说过，谁处理了不知道。

现在是：

> 哪条消息、生成了什么对象、谁审批、有没有发送，都能查。

## 9.3 降低 AI 外发风险

AI 不直接乱发。

默认逻辑是：

> AI 生成草稿，人来确认，系统留痕。

这比“让机器人直接回客户”更适合企业。

## 9.4 帮团队沉淀知识

客户和用户反复问的问题，会变成知识缺口。

后续可以沉淀成知识库，减少重复解释。

## 9.5 给管理者看日报和过程数据

老板不需要翻群聊。

可以看：

- 今天多少消息。
- 多少工单。
- 多少线索。
- 多少风险。
- 多少未回复问题。
- 哪些任务还没完成。

## 11. 如果给老板做演示，建议怎么讲？

可以这样讲：

> 我们这个系统不是再做一个聊天机器人，而是把公司每天散落在飞书、企微、钉钉里的业务消息变成可管理的流程。

然后演示：

1. 导入一批聊天记录，或者让飞书发一条真实消息。
2. 到“消息事件”看消息已经进来了。
3. 看系统把它识别成销售线索、工单、社群风险或招聘候选人。
4. 到对应业务页面看对象已经生成。
5. 到审批队列看 AI 回复草稿。
6. 人工批准后再发送。
7. 到运行日志看系统为什么这么判断。
8. 到知识库看问题如何沉淀。
9. 到报告中心看业务日报。

老板最容易理解的一句话：

> 以前群聊是信息黑洞，现在每条重要消息都能变成任务、线索、工单、审批和报告。
