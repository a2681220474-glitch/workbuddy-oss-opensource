## 第四章：一条消息如何完成全链路处理

时间：`02:35 - 04:10`

### 画面

以真实示例消息为主线：

```text
系统登录不上，一直报错，客户很着急。
```

依次展示：

- 飞书消息。
- 消息事件详情。
- 意图、风险和目标 Agent。
- AgentRun。
- 新生成的高优先级工单。
- 知识检索结果。
- 审批队列中的 AI 草稿。
- 审批通过。
- 飞书完成态卡片。
- 操作审计。

### 旁白

我们来看一条完整链路。客户在飞书中发送：“系统登录不上，一直报错，客户很着急。”消息进入 WorkBuddy 后，首先被标准化并去重。Agent Router 结合规则和模型判断，这是一条客服故障消息，风险较高，应交给客服工单 Agent。

系统随后创建一条高优先级 Ticket，记录客户、来源消息、处理状态和 SLA。同时，知识检索服务会从已发布知识中寻找登录故障的解决方案，并向 AI 草稿提供来源引用。

AI 会生成一份建议回复，但不会直接发送给客户。草稿首先进入审批队列。审批人可以查看原始消息、工单上下文、风险等级和知识引用，然后选择通过、编辑后通过，或者拒绝。

审批通过后，系统根据会话策略进行模拟发送或真实发送。飞书互动卡片会更新为已通过，并保留查看详情。发送结果、平台消息 ID、操作人和处理时间全部进入审计记录。至此，一条普通聊天消息完成了从发现问题到正式处理的业务闭环。

## 第五章：四大场景 Agent

时间：`04:10 - 05:45`

### 画面

四个场景依次出现，每个约 20 秒：

1. 客服工单。
2. 销售线索。
3. 社群运营。
4. 招聘入职。

### 旁白

围绕企业最常见的聊天场景，WorkBuddy OSS 提供四类核心 Agent。

客服工单与知识 Agent，可以识别故障、投诉、退款和产品问题，生成工单、判断优先级、跟踪 SLA，并把未知问题沉淀为知识缺口。

销售线索 Agent，可以从询价、合作和采购消息中识别潜在客户，记录兴趣点和评分，提示下一步，并推动线索从新线索、已联系、方案、谈判一直走向赢单或输单。

私域社群运营 Agent，会持续整理群活跃度、高意向用户、未回复问题、风险消息和社群任务。运营人员可以针对风险消息生成回复草稿，并送入审批。

招聘与入职 Agent，可以从简历和招聘沟通中提取候选人，完成岗位匹配分析，生成面试问题，并管理从筛选、面试、Offer 到入职的 Checklist。

这四类 Agent 并不是孤立页面，它们都共享同一套消息、任务、审批、知识和审计能力。

## 第六章：任务、知识和报告

时间：`05:45 - 06:55`

### 画面

- 跟进任务页：负责人、截止时间、逾期和状态。
- 知识缺口被采纳为知识条目。
- 知识混合检索、引用与反馈。
- 版本历史和回滚。
- 报告中心的六类报告。

### 旁白

Agent 的建议最终需要变成真实行动。跟进任务记录负责人、优先级、截止时间和关联业务对象，成员可以认领和推进任务，管理者可以看到逾期和无人处理事项。

在知识库中，系统会自动收集反复出现但没有标准答案的问题。团队可以把知识缺口采纳为正式条目，编辑、审核、发布和定期复审。检索同时结合关键词、向量相似度和知识质量，返回证据片段和引用。用户还可以评价结果是否有帮助，低质量内容会自动进入复审。每次修改都有版本记录，可以随时回滚。

报告中心则把消息和业务对象汇总为业务运营、客服、销售、社群、招聘和知识缺口报告，让管理者不再依赖人工整理日报。

## 第七章：工作台、审批和可追溯治理

时间：`06:55 - 07:50`

### 画面

- 工作台统计与“我的工作台”。
- 风险聚合、我的待办、我的审批。
- 审批详情和发送预检。
- 操作审计。
- Agent 运行日志。

### 旁白

日常使用从工作台开始。成员可以看到自己的待办、逾期和审批，管理者可以看到四大 Agent 的对象数量、风险和积压。

审批队列是产品最重要的安全边界。所有面向客户、候选人或社群成员的 AI 回复，默认先由人确认。价格、退款、合同、Offer、隐私和低置信度内容尤其不能绕过审批。

系统还分别记录人员操作和 AI 运行过程。操作审计回答“谁在什么时间做了什么”，AgentRun 则回答“AI 使用了什么输入、模型和 Prompt，为什么生成这个结果”。这让整套自动化保持可解释、可追踪和可复盘。

### 旁白

WorkBuddy OSS 可以部署在企业自己的 Ubuntu 服务器中。正式部署使用 Docker Compose，包含 Web、API、PostgreSQL、Redis、后台任务、消息 Worker 和 Caddy HTTPS。

公网只需要开放 SSH、HTTP 和 HTTPS，数据库、Redis 和内部 API 不应直接暴露。企业首先关闭真实外发，完成健康检查、登录初始化、模型测试、渠道接收和模拟发送，再在测试会话中逐步开启真实发送。

系统还提供数据库备份、校验和 PostgreSQL 隔离恢复演练。恢复测试会在临时数据库中进行，不会破坏正式业务数据。企业因此可以在保留数据控制权的同时，建立一套能够长期运行和恢复的私有 AI 工作流。
